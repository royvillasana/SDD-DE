# SDD-DE — Agent Configuration

This file is the **single source of truth** for all AI agents working in this project.
It applies to Claude Code, Cursor, Codex, Gemini, and any other AI coding assistant.

---

## 1. Core Principles

Apply these rules in every session, regardless of task size or complexity:

- **Spec-first, always**: Never implement from a brief alone. The Component Spec, Interaction Spec, and Page Spec must exist and be confirmed before any code is written.
- **One task at a time**: Execute one spec task per session. Mark it complete (`- [ ]` → `- [x]`) before moving to the next.
- **Token-referenced output**: Every color, spacing, radius, and font value in generated code must reference a design token variable. Hardcoded hex or raw pixel values fail review.
- **Pixel-accurate by default**: After every implementation, compare the live component to the design spec. Unresolved discrepancies block the Verify step.
- **Accessible by default**: Semantic HTML and ARIA attributes are not optional — they are part of the spec.
- **Incremental changes**: Prefer small, focused prompts over large "build the whole page" requests. One component, one task, one prompt.
- **Question assumptions**: If the spec is ambiguous, ask before implementing. An assumption that contradicts the spec wastes a full cycle.

---

## 2. Language Standards

- All spec documents, code comments, commit messages, and AI responses must be written in **English**.
- Component names follow **PascalCase**: `ButtonPrimary`, `HeroSection`, `PricingCard`.
- Design token variables follow **kebab-case** with semantic grouping: `--color-brand-primary`, `--spacing-4`, `--radius-lg`.
- File names follow **kebab-case**: `button-primary.tsx`, `hero-section.vue`, `price-card.svelte`.
- Git branch names: `feature/[component-name]-spec` or `feature/[page-name]-layout`.

## 3. Project Configuration

Before starting any SDD cycle, read `.sdd-de/project.yaml`.
If the file does not exist, run `/setup` first.

```yaml
# .sdd-de/project.yaml — generated by /setup or npx sdd-de
framework:              # react | next | vue | nuxt | svelte | sveltekit | angular | astro | vanilla
language:               # typescript | javascript
styling:                # css | css-modules | scss | tailwind | styled-components | emotion
design_source:          # figma | library | github | zip
figma_file_url:         # (design_source: figma) Figma file URL
figma_token_collection: # (design_source: figma) variable collection name
component_library:      # (design_source: library) shadcn | radix | mui | antd | chakra | mantine | headlessui | other
github_repo_url:        # (design_source: github) https://github.com/org/repo
github_branch:          # (design_source: github) branch name, e.g. main
github_component_dir:   # (design_source: github) path inside repo, e.g. src/components
zip_file_path:          # (design_source: zip) path to .zip archive
zip_component_dir:      # (design_source: zip) path inside zip, e.g. src/components
token_file:             # path to design token file
component_dir:          # root component directory
test_runner:            # vitest | jest | playwright | cypress | none
```

All generated code, file paths, imports, and framework idioms must match these values.
See `docs/framework-config.md` for framework-specific guidance.

## 4. Design System Source — Two Flows

`design_source` in `project.yaml` determines which workflow every skill uses:

| `design_source` | Source of truth | Key tool |
|---|---|---|
| `figma` | Figma file — frames, variables, component specs | Figma MCP |
| `library` | Component library (shadcn/ui, MUI, etc.) + brand tokens | Library docs + token file |
| `github` | GitHub repo containing component source files | WebFetch + GitHub API + Read |
| `zip` | ZIP archive containing component source files | Read (after extraction) |

Every skill (`enrich-brief`, `generate-artifacts`, `visual-verify`, `sync-tokens`) reads `design_source` and follows the appropriate branch. Never mix the two flows.

## 6. Skills

Skills live in `.sdd-de/ai-specs/skills/`. When a request matches a skill name, load and follow `SKILL.md` automatically before responding. Never guess at skill behavior — always read the SKILL.md.

| Slash Command | Purpose |
|---|---|
| `/setup` | Configure the project: framework, language, design system source |
| `/enrich-brief` | Transform a vague brief into a spec-ready story |
| `/generate-artifacts` | Generate all 3 spec files from the enriched story |
| `/visual-verify` | Run structured visual QA comparing live implementation to design spec |
| `/adversarial-review` | Red-team implementation before committing |
| `/sync-tokens` | Sync design tokens between design source and code |
| `/commit` | Commit with spec as PR description |
| `/sync-agent-symlinks` | Repair broken symlinks across `.claude/` and `.cursor/` |

---

## 7. Planning Model Requirement

For planning-heavy skills (`/setup`, `/enrich-brief`, `/generate-artifacts`), prefer the most capable available model (Opus or equivalent) to maximize reasoning quality on spec creation. Return to the standard model (Sonnet or equivalent) for implementation tasks.

---

## 8. Symlink Integrity and Multi-Agent Portability

The canonical source of truth for skills and agents is `ai-specs/`. Editor-specific paths (`.claude/skills/`, `.cursor/skills/`) reference the canonical source via symlinks.

- Broken symlinks = skill is silently unavailable
- Run `/sync-agent-symlinks` whenever a skill is added, renamed, or removed
- Symlinks must use relative paths for portability across machines

---

## 9. Mandatory Spec Update Rule

When a change request arrives **after Apply but before Commit**:
1. Update the affected spec artifact first — never implement a change without updating the spec
2. Add the new task to the spec task list
3. Implement only after the spec reflects the change
4. Re-run `/visual-verify` and `/adversarial-review` before committing
